# 风控信贷策略 Skills 设计框架

> 版本: v0.3 | 日期: 2026-06-16 | 状态: 待审核

---

## 一、设计目标

构建一个**用户问答驱动**的风控策略 skills 框架，覆盖从规则挖掘到策略上线的完整链路。与 A 卡建模 skills 的线性流水线不同，策略 skills 的核心是**可独立调用的分析方法论工具箱**，每次使用前必须由用户明确产品形态、还款方式和应用场景，禁止自动推断。

---

## 二、产品定义（用户问答驱动）

### 2.1 强制询问流程

每次使用 skills 时，**必须依次询问用户以下三个问题**，任一未明确则**终止执行**：

```
Q1: 请确认产品形态？
    [1] 循环贷
    [2] 非循环贷
    [3] 其他（请描述）

Q2: 请确认还款方式？
    [1] 等额还款
    [2] 一次性还本付息
    [3] 其他（请描述）

Q3: 请确认应用场景？
    贷前:
      [A1] 首借
      [A2] 复借
      [A3] 额度策略
      [A4] 定价策略
      [A5] 支用策略
    贷中:
      [B1] 提额策略
      [B2] 降额策略
      [B3] 提价策略
      [B4] 降价策略
      [B5] 预警策略
```

### 2.2 产品 2×2 矩阵（参考卡片，非自动路由）

| | 等额还款 | 一次性还本付息 |
|---|---|---|
| **循环贷** | 有额度、多次支用、按月还本付息 | 有额度、多次支用、到期还本 |
| **非循环贷** | 一次性放款、按月还本付息 | 一次性放款、到期还本 |

### 2.3 产品形态对策略的影响（参考）

| 产品特征 | 策略影响 |
|----------|----------|
| 循环贷 | 有额度管理概念（提额/降额）、有支用审批环节 |
| 非循环贷 | 无额度管理、复借=重新申请；支用审批仅在授信有效期内触发 |
| 等额还款 | 关注月供负担/DTI，现金流稳定性 |
| 一次性还本付息 | 关注到期违约概率，还款能力集中评估 |

### 3.1 场景定义

#### 贷前

| 场景 | 适用产品 | 核心决策 |
|------|----------|----------|
| **首借** | 全部 | 是否通过？给多少额度？定什么价？ |
| **复借** | 非循环贷 | 再次申请时，有历史还款记录，如何调整策略？ |
| **额度策略** | 全部 | 初始额度怎么定？（循环贷=授信额度，非循环贷=放款金额） |
| **定价策略** | 全部 | 利率/费率怎么定？风险定价 vs 监管上限 vs 市场竞争 |
| **支用策略** | 全部 | 客户提款时是否重新审查？（循环贷：每次支用；非循环贷：授信有效期后支用） |

#### 贷中

| 场景 | 适用产品 | 核心决策 |
|------|----------|----------|
| **提额策略** | 循环贷 | 好客户→更高额度，触发条件+提额幅度 |
| **降额策略** | 循环贷 | 风险恶化→降低额度，触发条件+降额幅度 |
| **提价策略** | 循环贷 | 风险上升→提高利率，触发条件+提价幅度 |
| **降价策略** | 循环贷 | 优质客户→降低利率（留存竞争/行为优异） |
| **预警策略** | 全部 | 提前识别风险信号→管控措施（冻结/降额/人工审核） |

## 四、数据模式

场景确认后，询问用户当前样本的数据模式：

| 模式 | 说明 | 可用的分析方法 |
|------|------|---------------|
| **纯规则** | 无模型分，仅特征+标签 | 单规则挖掘、CART规则挖掘、策略模拟 |
| **有模型分** | 样本包含 A/B/C 卡分数 | 以上全部 + 分数截断优化 + 分数×规则联合策略 |
| **混合** | 部分特征+模型分 | 全部可用，但需注意分数缺失样本的处理 |

## 五、分析流程（核心）

策略分析按以下流程组织，每个模块可独立调用：

```
Score Cutoff 优化 ──→ 规则挖掘 ──→ 策略模拟 ──→ Swap in/out 分析 ──→ 上线部署
```

### 5.1 分数截断优化（有模型分时可用）

给定模型分数和标签，输出最优截断点：

| 指标 | 含义 |
|------|------|
| 各截断点下的通过率、坏账率 | 遍历分数分位数作为候选截断点 |
| KS 最大点 | 区分能力最强的截断点 |
| 坏账率约束下的最优截断 | 给定容忍坏账率上限 → 最大化通过率 |
| 通过率约束下的最优截断 | 给定最低通过率 → 最小化坏账率 |
| **效率前沿曲线** | 通过率 vs 坏账率 trade-off 曲线，前沿上的点代表 Pareto 最优 |

### 5.2 规则挖掘

#### 5.2.1 单规则挖掘

对单个特征变量，遍历所有可能阈值，计算：

| 指标 | 公式 | 含义 |
|------|------|------|
| **count** | `N_hit` | 命中规则的样本数 |
| **hit_rate** | `N_hit / N_total` | 命中率 |
| **bad_rate** | `N_bad_hit / N_hit` | 命中样本的逾期率 |
| **passed_bad_rate** | `N_bad_pass / (N_total - N_hit)` | 未命中样本的逾期率 |
| **lift** | `bad_rate / overall_bad_rate` | 提升度（>1 表示命中样本坏账率高于整体） |
| **bad_capture** | `N_bad_hit / N_bad_total` | 坏客户捕获率 |

输出：每个变量的最优阈值推荐（按 lift 排序、按 bad_capture 排序、按 hit_rate 约束下的 bad_rate 排序）。

#### 5.2.2 CART 决策树规则挖掘

用 CART 决策树从多维度发现规则组合：

- 输入：特征矩阵 + 标签
- 超参：max_depth, min_samples_leaf, class_weight
- 输出：每条叶子节点路径 → 一条规则（多条件 AND 组合）
- 每条规则同样计算 count / hit_rate / bad_rate / lift / bad_capture

**方法论要点**：

- **规则可解释性约束**：max_depth 建议 ≤ 4，min_samples_leaf 建议 ≥ 总样本的 1%，确保规则简洁可落地
- **叶子节点合并**：bad_rate 相近的叶子（差异 < 1% 或统计不显著）合并为同一规则，减少规则数量
- **单调性验证**：对已知单调特征（如逾期次数越多风险越高），验证规则方向是否与业务逻辑一致，不一致时标记警告
- **过拟合检测**：对比 train 和 test 的 lift 值，lift 下降 > 20% 表示过拟合，需增大 min_samples_leaf 或减小 max_depth
- **扩展：随机森林规则提取**（可选）：从多棵 CART 中提取高频规则，按出现频率排序，取 Top-N 作为候选规则

#### 5.2.3 规则组合

将多条单规则按逻辑组合：

- **AND 组合**（交集）：多条规则同时命中 → 更严格，bad_rate 更高但 hit_rate 更低
- **OR 组合**（并集）：任一规则命中 → 更宽松，hit_rate 更高但 bad_rate 更低
- **级联**（sequential）：规则按优先级执行，前一条通过才进入下一条
- **路由**：不同客群走不同规则集（如：高分段规则A，低分段规则B）

---

### 5.3 策略模拟

给定一组规则（含截断点），模拟策略上线后的效果：

| 指标 | 含义 |
|------|------|
| **通过率** (approval_rate) | 未被任何规则拒绝的比例 |
| **人维度逾期率** | 逾期人数 / 通过人数 |
| **金额维度逾期率** | 逾期金额 / 放款总额 |
| **拒绝人群中坏客户占比** | 拒绝的准确性 |
| **通过人群中坏客户占比** | 漏网率 |

支持两种视角：
- **通过率约束**：给定目标通过率，反推最优截断
- **坏账率约束**：给定容忍坏账率，反推最大通过率

**效率前沿**：将多组策略参数（不同截断/不同规则组合）的（通过率, 坏账率）绘制为散点图，前沿上的点代表 Pareto 最优——即在给定通过率下坏账率最低的策略。新策略应尽量落在或接近前沿。

---

### 5.4 Swap-in & Swap-out 分析

对比新旧策略的效果差异，回答"替换这套策略，效果变好还是变差？"。

#### 5.4.1 分析框架

```
          ┌──────────────────────────────────┐
          │  全量样本（有 Y_label）            │
          │                                  │
          │  ┌─────────────┐  ┌─────────────┐│
          │  │ Out 策略命中 │  │ Out 策略通过 ││
          │  │ (已有风险表现)│  │ (已有风险表现)││
          │  └──────┬──────┘  └──────┬──────┘│
          │         │                │       │
          │   直接对比 bad_rate 即可    │       │
          │                           │       │
          │  ┌──────────────────────────────┐ │
          │  │  In 策略（新策略）             │ │
          │  │  In 命中 = 新增拒绝/新增通过   │ │
          │  │  需要预估这部分客户的风险表现   │ │
          │  └──────────────────────────────┘ │
          └──────────────────────────────────┘
```

#### 5.4.2 Swap-out 分析（已有风险表现，直接对比）

Out 策略命中的客户有历史风险标签，可直接计算：

| 对比维度 | 指标 |
|----------|------|
| 通过率变化 | old_approval_rate → new_approval_rate |
| 人维度逾期率变化 | old_bad_rate → new_bad_rate |
| 金额维度逾期率变化 | old_amount_bad_rate → new_amount_bad_rate |
| 拒绝重叠矩阵 | 新旧同时拒绝 / 仅旧拒绝 / 仅新拒绝 / 均通过 |
| 仅旧拒绝人群的 bad_rate | 旧策略拒绝但新策略通过的人群，实际逾期率 |

#### 5.4.3 Swap-in 分析（新策略命中人群，需要预估）

新策略命中的人群中，有一部分在旧策略下通过了（有风险标签），有一部分在旧策略下也被拒绝了（无独立风险表现）。预估方案待单独讨论确定。

#### 5.4.4 分路由 Swap

在场景明确的前提下，按不同路由（渠道/分段/产品等）独立计算 Swap 效果：

- 每个路由下独立计算通过率/逾期率变化
- 识别"策略在哪个路由下失效/恶化"
- 输出：路由级 Swap 对比表 + 整体加权汇总

---

### 5.5 策略上线

- **灰度上线**：X% 流量走新策略，逐步放量
- **A/B 测试**：实验组 vs 对照组，统计显著性检验
- **全量切换**：一步到位（需审批确认）

> 注：上线后监控（通过率/逾期率趋势/PSI）通常在 BI 系统中完成，本 skills 不覆盖持续监控功能。

## 七、核心指标定义

### 7.1 人维度逾期率

```
人维度逾期率 = 逾期人数 / 通过总人数
```

- 分子：通过审批后发生逾期（通常 30+/60+/90+ DPD）的**人数**
- 分母：策略审批通过的**人数**
- 适用于：首借、复借、支用等审批场景

### 7.2 金额维度逾期率

```
金额维度逾期率 = 逾期本金余额 / 放款总额
```

- 分子：逾期客户对应的**未偿还本金金额**
- 分母：策略审批通过后**实际放款的总金额**
- 适用于：需要评估资金风险的场景（大额客户逾期影响更大）

### 7.3 单规则指标

| 指标 | 公式 | 含义 |
|------|------|------|
| **count** | `N_hit` | 命中规则样本数 |
| **hit_rate** | `N_hit / N_total` | 命中率 |
| **bad_rate** | `N_bad_hit / N_hit` | 命中样本逾期率 |
| **passed_bad_rate** | `N_bad_pass / (N_total - N_hit)` | 未命中样本逾期率 |
| **lift** | `bad_rate / overall_bad_rate` | 提升度，>1 表示规则有效 |
| **bad_capture** | `N_bad_hit / N_bad_total` | 坏客户捕获率 |

### 7.4 分数截断指标

| 指标 | 含义 |
|------|------|
| 各截断点 (通过率, 坏账率) | 遍历分数分位数，计算每个截断点的通过率和坏账率 |
| KS 最大点 | 通过率和坏账率之差最大的截断点 |
| 效率前沿 | 所有截断点中 Pareto 最优的集合 |

### 7.5 Swap 对比指标

| 指标 | 含义 |
|------|------|
| Δ approval_rate | 新旧通过率之差 |
| Δ bad_rate（人维度） | 新旧人维度逾期率之差 |
| Δ bad_rate（金额维度） | 新旧金额维度逾期率之差 |
| overlap_reject | 新旧同时拒绝的样本量 |
| old_only_reject | 仅旧策略拒绝的样本量 |
| new_only_reject | 仅新策略拒绝的样本量 |
| old_only_bad_rate | 仅旧策略拒绝样本的实际逾期率（有标签，可直接算） |
| new_only_bad_rate | 仅新策略拒绝样本的逾期率（需预估） |

## 八、数据契约约定

### 8.1 输入数据必需列

| 列名 | 类型 | 说明 |
|------|------|------|
| `sample_id` | str/int | 样本唯一标识 |
| `Y_label` | int (0/1) | 逾期标签 |
| `sample_date` | datetime | 样本时间（用于 OOT/监控时间窗口） |
| `amount` | float | 放款金额（用于金额维度逾期率计算） |

### 8.2 可选列

| 列名 | 类型 | 说明 |
|------|------|------|
| `model_score` | float | 模型分（A/B/C 卡） |
| `route` | str | 路由标签（渠道/分段/产品等），分路由分析时必需 |
| `sample_weight` | float | 样本权重 |

### 8.3 保护列

以下列**禁止作为规则特征**：`sample_id`, `Y_label`, `sample_date`, `amount`（金额仅用于指标计算，不作为特征）。

### 8.4 标准化输出 Schema

各分析模块的输出遵循统一 schema，确保模块间可串联。

```python
# 单规则挖掘输出
@dataclass
class SingleRuleResult:
    feature: str
    direction: str        # '<=' or '>'
    threshold: float
    count: int
    hit_rate: float
    bad_rate: float
    passed_bad_rate: float
    lift: float
    bad_capture: float

# 分数截断输出
@dataclass
class ScoreCutoffResult:
    cutoff: float
    approval_rate: float
    bad_rate_person: float
    bad_rate_amount: float
    ks: float
    on_frontier: bool     # 是否在效率前沿上

# 策略模拟输出
@dataclass
class SimulationResult:
    rules: list[SingleRuleResult]
    approval_rate: float
    bad_rate_person: float
    bad_rate_amount: float
```
